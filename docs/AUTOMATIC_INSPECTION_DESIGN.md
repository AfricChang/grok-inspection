# 自动巡检设计方案

状态：第一版已实现，待运行环境验收

本文档描述 Grok Inspection CPA 插件自动巡检的产品和技术方案，并记录第一版实现后的实际接口与验收边界。

## 1. 目标与边界

自动巡检应在 CPA 进程内运行，不依赖浏览器页面打开。整体采用“健康账号被动检测、问题账号定时主动复检”的混合模式：健康账号不再通过定时探测产生额外上游请求，而是在真实业务请求产生 Usage 失败记录时更新状态；权限、额度和异常账号继续按规则定时复检。

自动化边界：

- 允许自动执行 `enable` 和 `disable`。
- 永远禁止自动执行 `delete`。
- `需重登` 账号不进入任何自动巡检任务。
- `异常` 结果只记录和复检，不因一次异常自动禁用账号。
- 健康账号不进入定时主动巡检；通过 CPA 实际业务请求被动检测。
- 手工巡检和手工删除仍保留现有能力，不受自动化限制影响。

## 2. 核心概念

### 2.1 巡检规则

规则决定什么时候触发，以及优先选择哪些分类的账号。建议字段：

```json
{
  "id": "rule-problem-recheck",
  "name": "问题账号复检",
  "enabled": true,
  "weekdays": [1, 2, 3, 4, 5, 6, 7],
  "window_start": "00:00",
  "window_end": "23:59",
  "interval_minutes": 120,
  "scope": ["permission_denied", "quota_exhausted"],
  "workers": 6,
  "include_disabled": true,
  "apply_recommendations": true
}
```

字段约束：

- `window_start`、`window_end` 使用 CPA 所在机器的本地时间。
- 开始时间等于结束时间表示全天窗口。
- 结束时间早于开始时间表示跨午夜窗口，例如 `22:00` 到 `06:00`。
- `interval_minutes` 建议最小 15 分钟，并设置合理上限。
- `scope` 支持 `all`、`healthy`、`permission_denied`、`quota_exhausted`、`other`。
- 自动任务不提供 `reauth` 选项；历史结果为 `reauth` 的账号统一排除。

### 2.2 巡检范围与结果分类

需要区分“本次选择哪些账号巡检”和“巡检结果属于什么分类”：

- `all`：选择所有满足条件的问题账号；不包含健康账号，因为健康账号通过 Usage 被动检测。
- `healthy`：保留为手工巡检范围，第一版不作为自动规则范围。
- `permission_denied`：复查权限是否恢复。
- `quota_exhausted`：复查额度是否恢复。
- `other`：复查模型不可用、探测异常、未知等非主分类。

分类规则只根据上一次已保存结果选取账号。`all` 仍可作为手工巡检范围，自动规则建议直接选择问题分类；新账号和健康账号通过手工巡检或真实业务请求进入状态管理。

### 2.3 健康账号被动检测

CPA 当前版本支持插件 `UsagePlugin`。xAI 执行器在真实请求完成后，会向插件发送 Usage 记录；成功和失败请求都会发送。失败记录包含 `Provider`、`AuthID`、`AuthIndex`、HTTP 状态码和错误正文，因此插件可以准确定位产生错误的 Grok 账号。

健康账号不再通过自动规则主动发起探测。普通 HTTP 429、5xx、超时和连接错误不能直接判定为额度用尽；只有包含 Grok 明确免费额度耗尽标识的失败记录才能归类为 `quota_exhausted`。Usage 回调必须快速返回，账号状态写入和禁用操作异步执行，不能阻塞真实业务请求。

自动禁用优先复用已认证管理请求中的 Management Key（仅进程内存缓存，不落盘）；如果 CPA 进程配置了 `MANAGEMENT_PASSWORD` 或 `CPA_MANAGEMENT_KEY`，则作为无页面场景的后备来源。两者都没有时仍记录识别结果，但自动禁用会记录失败原因。

CPA 可能在一个请求中先后尝试多个账号。插件必须按每条 Usage 记录的 `AuthIndex` 独立处理，不能只根据最终请求是否成功判断账号状态。

### 2.4 问题账号冷却时间

规则的触发间隔不等于每个账号的实际探测间隔。每个问题账号还要经过冷却时间过滤，防止同一账号被高频重复探测。

默认冷却策略：

| 上次分类 | 默认冷却时间 | 自动巡检处理 |
|---|---:|---|
| 健康 | 不适用 | 通过真实业务请求被动检测 |
| 权限被拒 | 2 小时 | 高频确认是否恢复 |
| 额度用尽 | 2 小时 | 高频确认是否恢复 |
| 异常、模型不可用、未知 | 30 分钟 | 短周期重试 |
| 需重登 | 永久 | 自动跳过 |

冷却时间应先作为系统默认值实现，后续可以再开放为全局配置。每次真正发起探测后更新 `last_inspected_at` 和 `next_inspect_at`。分类变化后，按新分类重新计算下次可巡检时间。

健康账号不会因为自动巡检规则产生额外探测请求；权限、额度和异常账号则可以按各自周期复检。

## 3. 调度行为

每次调度触发时执行以下流程：

```text
到达规则时间点
  → 判断是否在时间窗口内
  → 检查是否已有手工巡检、自动巡检或批量操作
  → 根据规则 scope 选择问题账号
  → 排除需重登账号
  → 按账号 next_inspect_at 过滤冷却中的账号
  → 合并其他同时触发规则，按稳定身份去重
  → 调用现有 inspectionEngine.start()
  → 巡检完成后筛选 enable/disable 建议
  → 执行安全建议操作（禁止 delete）
  → 保存执行摘要和下次调度状态
```

并发和冲突处理：

- 同一账号在同一轮只探测一次。
- 手工任务正在运行时，自动任务不并发启动。
- 当前时间点无法执行时，最多延后到 CPA 空闲后执行一次；超过一个间隔则跳过，等待下一次计划。
- CPA 重启后，如果错过的时间点不超过一个间隔，允许补跑一次；否则不追赶历史任务。
- 插件卸载时停止调度器，但不强制中止正在进行的探测或批量操作。

## 4. 建议的实际规则

### 4.1 问题账号主动复检规则

```text
每天 00:00～23:59，每隔 2 小时
范围：权限被拒、额度用尽
自动操作：启用/禁用
```

用途是确认已经发现的问题账号是否恢复。健康账号不通过这条规则主动探测；它们在真实业务请求返回额度或权限错误时被动转为问题账号。

### 4.2 异常重试规则

```text
每天 08:00～23:00，每隔 30 分钟
范围：异常
自动操作：关闭
```

异常可能是短暂网络或上游问题，只复检，不根据一次异常禁用账号。恢复为健康后，账号退出自动主动巡检，回到 Usage 被动检测状态。

## 5. 自动建议操作安全策略

自动模式只允许以下动作：

| 检测结果 | 当前状态 | 自动动作 |
|---|---|---|
| 健康 | 已启用 | 不操作 |
| 健康 | 已禁用 | 启用 |
| 权限被拒 | 已启用 | 禁用 |
| 权限被拒 | 已禁用 | 不操作 |
| 额度用尽 | 已启用 | 禁用 |
| 额度用尽 | 已禁用 | 不操作 |
| 异常 | 任意 | 不操作 |
| 需重登 | 任意 | 跳过，并记录禁止删除 |

被动 Usage 处理规则：

- 明确的 Grok 免费额度耗尽标识：标记为 `quota_exhausted`。
- 明确的权限拒绝或受限标识：标记为 `permission_denied`。
- 401、过期或无效 Token：标记为 `reauth`，之后自动跳过。
- 普通 429、5xx、超时、连接异常：标记为异常或保留原分类，不自动禁用。

禁止删除必须在三层实现：

1. UI 不提供自动删除选项。
2. 自动任务生成候选时只允许 `enable`、`disable`。
3. 后端执行器对自动任务的 `delete` 再次拒绝，即使配置文件被手工修改也不能删除账号。

## 6. 配置与运行记录

不修改现有结果文件结构，建议新增独立文件：

```text
data/grok-inspection/
├── results.json
├── automation.json
└── automation-history.json
```

`automation.json` 保存规则和调度器状态；账号自动巡检时间字段保存在 `results.json` 的账号结果中。`automation-history.json` 只保留最近约 50 次执行摘要，例如：

- 规则名称
- 计划时间和实际开始时间
- 巡检账号数
- 跳过数及原因
- 成功、失败的启用/禁用数量
- 被禁止的删除数量
- 错误摘要

Usage 被动检测事件建议单独记录最近摘要，包括账号、状态分类、状态码、错误码和是否已提交自动操作，避免把每次成功业务请求都写入历史文件。

文件使用临时文件加原子重命名方式保存，避免 CPA 异常退出时破坏配置。

建议在结果记录中增加：

```text
last_inspected_at
next_inspect_at
auto_inspect_excluded
auto_inspect_exclude_reason
```

## 7. 管理接口和页面

已实现的管理接口：

```text
GET    /automation
POST   /automation/rules       # body 中带 id 时更新，否则创建
DELETE /automation/rules?id=...
POST   /automation/run         # body: {"id":"..."}
GET    /automation/history
```

页面增加“自动巡检”区域，展示：

- 规则名称、时间窗口、间隔、范围、启用状态
- 下一次执行时间
- 最近一次执行摘要
- 立即执行、编辑、停用、删除规则

当自动任务运行时，顶部分类卡片、当前筛选、并发、检测进度和自动巡检状态必须跟随实际运行规则更新；自动任务完成检测进入建议操作阶段时，进度文案也要切换为自动操作进度，不能继续显示上一次手工巡检分类。

页面空闲时仍以低频轻量状态请求监听 `results_gen`。Usage 被动检测写入新分类或自动禁用更新账号状态后，页面应自动拉取完整结果，实时更新顶部分类数量和账号列表；同时刷新最近记录，输出“检测到额度用尽”“自动禁用成功”或具体失败原因。

编辑规则时明确提示：自动操作只包含启用和禁用，永不自动删除账号；需重登账号不会进入自动巡检。

## 8. 实现拆分与验收

实现已按以下顺序完成：

1. 增加规则模型、校验和独立落盘。
2. 增加调度器生命周期，并接入插件初始化、重配置和关闭流程。
3. 声明 `UsagePlugin` 能力，实现 `usage.handle` 和 xAI 失败记录分类。
4. 增加账号冷却字段与问题账号候选筛选逻辑。
5. 复用现有巡检引擎完成问题账号主动复检，避免实现第二套探测流程。
6. 增加自动建议操作的 `enable/disable` 白名单和 `delete` 双重拦截。
7. 增加自动巡检管理接口和页面。
8. 补充 Usage 失败分类、账号身份映射、时间窗口、冷却、冲突、重启补跑和禁止删除测试。

## 9. 第一版明确不做

- 完整 Cron 表达式。
- 健康账号定时主动轮询。
- 自动删除或自动清理 Auth 文件。
- 失败无限重试。
- 邮件、Webhook 等外部通知。
- 复杂条件表达式和账号标签规则。
